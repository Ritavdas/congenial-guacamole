Building a Design System from Scratch with Radix and Tailwind

✨ AI Summary

This article walks through building a production-grade design system using Radix UI primitives for accessibility and Tailwind CSS for styling. Key takeaways: (1) Start with design tokens mapped to CSS custom properties, (2) Use Radix for complex interactive patterns like dialogs, popovers, and dropdowns, (3) Build a variant system using class-variance-authority, and (4) Document components with Storybook for team adoption.

Design systems are the backbone of scalable product development. They ensure consistency, speed up development, and create a shared language between designers and engineers. But building one from scratch can feel overwhelming.

In this guide, we'll walk through creating a production-grade design system using Radix UI for accessible primitives and Tailwind CSS for a utility-first styling approach. By the end, you'll have a system that's both flexible and maintainable.

Why Radix + Tailwind?

The combination of Radix and Tailwind has become the de facto standard for modern React design systems. Here's why:

"The best design systems are invisible. They should feel like a natural extension of the platform, not a framework imposed on top of it."
— Josh Comeau

Radix provides unstyled, accessible primitives that handle all the complex interaction patterns — focus management, keyboard navigation, screen reader announcements — while Tailwind lets you apply your visual design language without fighting CSS specificity wars.

Setting Up Design Tokens

Every design system starts with tokens. These are the atomic values that define your visual language: colors, spacing, typography, shadows, and radii.

// tokens.ts
export const tokens = {
  colors: {
    primary: { 50: '#eef2ff', 500: '#6366f1', 900: '#312e81' },
    neutral: { 50: '#fafafa', 500: '#737373', 900: '#171717' },
  },
  spacing: { xs: '4px', sm: '8px', md: '16px', lg: '24px', xl: '32px' },
  radii: { sm: '6px', md: '8px', lg: '12px', full: '9999px' },
};

Mapping Tokens to CSS Custom Properties

The key insight is mapping these tokens to CSS custom properties in your Tailwind config. This gives you the best of both worlds: type-safe tokens in JavaScript and utility classes in your templates.

Building Your First Component

Let's build a Button component that demonstrates the full pattern: Radix primitives, CVA for variants, and Tailwind for styling.

import { cva, type VariantProps } from "class-variance-authority";

const buttonVariants = cva(
  "inline-flex items-center justify-center rounded-lg font-medium transition-colors",
  {
    variants: {
      variant: {
        default: "bg-primary-500 text-white hover:bg-primary-600",
        outline: "border border-neutral-300 hover:bg-neutral-50",
        ghost: "hover:bg-neutral-100",
      },
      size: {
        sm: "h-8 px-3 text-sm",
        md: "h-10 px-4 text-sm",
        lg: "h-12 px-6 text-base",
      },
    },
    defaultVariants: { variant: "default", size: "md" },
  }
);

This pattern is incredibly powerful because it gives you type-safe variants that compose naturally with Tailwind's utility classes, while Radix handles all the accessibility concerns under the hood.
